Fix guard false "No active plan"

Anchor the PreToolUse guard to the repo root, and make index writes atomic, so it stops blocking legitimate edits.

created guard cwd atomic-write
Created 2026-07-03

Context

A team using the harness reported the PreToolUse guard blocking legitimate edits with "No active plan for this branch" even when a valid created plan for the current branch exists. Two independent root causes, both confirmed against the code.

Approach

Defect 1 — the guard trusted the payload cwd for resolving the repo root. When the agent's shell sits in a subdirectory (a monorepo package), hasActivePlan looked for <subdir>/docs/plans, found nothing, and denied. The same subdirectory cwd also made the relative-path check resolve to .. for files addressed from the repo root, which silently allowed edits to locked plans and the generated indexes. So the bug broke enforcement in both directions.

Fix: resolve the true repo root and measure paths from it. A new repoRoot(cwd) helper runs git rev-parse --show-cdup and resolves it against cwd — deliberately not --show-toplevel, which canonicalizes symlinks (for example /tmp to /private/tmp) and would reintroduce the mismatch against an agent-supplied path. hasActivePlan and toRel now classify from the repo root; toRel still resolves a relative file path against cwd, since that is what it is relative to. All four agent adapters share one guard, so no adapter changes are needed.

Defect 2 — the index rewrite was not atomic. build-index.mjs rewrote plan files with a truncate-then-write writeFileSync. A guard reading frontmatter during that write could catch a half-written file, see no HARNESS block, and count the plan as absent — a false deny. This is narrow (needs two concurrent processes and a plan-body change) but cheap to close.

Fix: a new writeFileAtomic helper writes to a temp file in the same directory and renames over the target (rename is atomic within a filesystem). It replaces every writeFileSync in the index builder.

Files

  • .harness/bin/lib/hook-io.mjs — add repoRoot(cwd).
  • .harness/bin/lib/guards.mjs — classify from the repo root in hasActivePlan and toRel.
  • .harness/bin/lib/fs-atomic.mjs — new atomic-write helper.
  • .harness/bin/build-index.mjs — use atomic writes at all three write sites.

Verification

Defect 1 (isolated temp repo, real checkEdit): a project file is allowed at both the repo-root and subdirectory cwd; a locked plan is denied at both. End-to-end through the Claude adapter, a generated index stays blocked even from a subdirectory cwd.

Defect 2 (concurrent writer and reader): plain writes produced ~2.3% null reads (1400 of 60000); atomic writes produced zero (0 of 116000).

node .harness/bin/build-index.mjs still rebuilds both index pages with no unexpected churn.
