Generated plan status

Generate each plan's status badge and dates line from frontmatter so resolving a plan can't leave the rendered page out of sync.

implemented status generated drift
Created 2026-06-28 · Implemented 2026-06-28

Context

A plan's state lives in two places that must agree: the <!--HARNESS ... state: ... --> frontmatter, which is the field the lock reads, and a hand-authored badge plus dates line in the HTML body. Resolving a plan is a single frontmatter edit that flips the state and locks the file in the same stroke, so the body badge is frozen at created forever. Every resolved plan ships a self-contradicting page. The fix removes the redundancy by making the visible status a generated artifact with one source of truth.

Approach

Wrap the badge and dates line in generator-owned marker comments (harness:status and harness:dates). Extend build-index.mjs with a syncPlanBodies() pass that rewrites those regions from each plan's frontmatter, reusing the existing stateBadge() helper and writing only when content changes. The builder already runs as plain Node through the rebuild-index.mjs PostToolUse hook, outside the Edit-tool guard, so it can rewrite a locked plan's badge directly with no second manual edit and no rebuild loop. The guard itself stays exactly as strict.

Files

.harness/templates/PLAN.template.html — wrap the badge and dates line in the markers. .harness/bin/build-index.mjs — add the readFileSync import, datesLine(), replaceRegion(), and syncPlanBodies(); call it from main(). .harness/instructions/planning.md — document that the badge and dates are generated and that the resolving edit must be atomic (state flip and date field in one edit).

Verification

Create a fresh plan from the updated template, run node .harness/bin/build-index.mjs, and confirm the badge and dates render from frontmatter. Flip the frontmatter to implemented with an implemented: date in one edit; the PostToolUse hook rebuilds and the badge and dates follow with no manual body edit. Confirm the resolved plan is still locked against any Edit-tool change, that no hand-authored state- class remains outside the markers, and that running the builder twice produces no further change.
